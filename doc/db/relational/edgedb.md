# EdgeDB / Gel — typed Postgres with a real query language

**Main link:** <https://www.geldata.com/>

Old domain (still works): <https://www.edgedb.com/> · Docs: <https://docs.geldata.com/> · Repo: <https://github.com/geldata/gel>

## Summary

EdgeDB is a database layer that runs on top of [[postgresql]] but exposes its own object/graph schema model and a typed query language called **EdgeQL** (with optional GraphQL/SQL access too). It's effectively "what you wish your ORM gave you, baked into the database". In late 2024 the project was rebranded **Gel** (the company is now Gel Data), though the EdgeDB name is still widely used. The repo and CLI binary moved to `gel`.

## Insight

Reach for EdgeDB/Gel when:

- You like Postgres but hate writing SQL by hand for object-shaped data with deep relationships, and you also dislike heavy ORMs.
- You want migrations, schema, and queries to be statically typechecked end-to-end, with first-class TypeScript / Python / Rust client codegen.
- You want a "schema as source code" workflow (`.esdl` / `.gel` schema files committed to git) rather than ALTER-statement migrations.

Honest caveats:

- It's still a younger ecosystem than plain Postgres + Prisma/SQLAlchemy/Diesel. Library coverage and StackOverflow answers are thinner.
- The 2024 rebrand from EdgeDB → Gel was disruptive: docs, package names, search hits, and binary names all changed. Expect some link rot.
- Underneath it's still Postgres, so if you ever need to bail out you can — but you'll inherit a schema that was generated by EdgeQL, not designed for direct SQL use.
- Compare with PostgREST + Postgres views (the "no client library, talk HTTP to your DB" approach), or with Prisma / Drizzle / SQLC if you'd rather stay closer to plain SQL.

## References / raw notes

> EdgeDB is a next-generation graph-relational database designed as a spiritual successor to the relational database. It inherits the strengths of SQL databases: type safety, performance, reliability, and transactionality. But instead of modeling data in a relational (tabular) way, EdgeDB represents data with object types containing properties and links to other objects. It leverages this object-oriented model to provide a superpowered query language that solves some of SQL's biggest usability problems.

The core unit of schema is the **object type**, analogous to a SQL table. Object types contain **properties** and **links** to other object types, forming a schema graph. EdgeQL queries traverse those links naturally instead of requiring explicit joins.
